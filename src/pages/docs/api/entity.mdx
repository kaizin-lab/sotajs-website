## `createEntity(config)`

Creates an **Entity** — an object with a unique identity (`id`), mutable state changed only through
defined actions, and computed (derived) properties.

```ts
import { z } from 'zod';
import { createEntity } from '@sotajs/ddd';

const UserSchema = z.object({
  id: z.string().uuid(),
  username: z.string(),
  bio: z.string().optional(),
  status: z.enum(['active', 'inactive']),
  deactivatedAt: z.date().optional(),
});

const User = createEntity({
  schema: UserSchema,
  actions: {
    deactivate(state) {
      state.status = 'inactive';
      state.deactivatedAt = new Date();
    },
  },
  invariants: [
    (props) => {
      if (props.status === 'inactive' && !props.deactivatedAt)
        throw new Error('Inactive user must have deactivatedAt');
    },
  ],
  computed: {
    displayName(props) { return props.username; },
  },
});
type User = ReturnType<typeof User.create>;

const user = User.create({ id: 'b3f1ae2c-abcd', username: 'alice', status: 'active' });
user.actions.deactivate();
console.log(user.displayName); // 'alice'
```

> **How does `displayName` work?** It's a getter defined on the entity prototype.
> Each computed key becomes a property — not a function call. Defined once
> in config, reads like native data. Recalculates from current props on every access.
> [Source: entity.ts](https://github.com/kaizin-lab/sotajs-ddd/blob/main/src/entity.ts)

## Config

| Field | Required | Description |
|---|---|---|
| `schema` | No* | Zod/Valibot schema — `.parse()` interface. Provides type inference + auto-setters. |
| `type` | No* | ArkType type — callable. Fastest validation path. |
| `validate` | No* | Generic `(data: unknown) => TProps`. Use with any validator or plain function. |
| `actions` | **Yes** | Named operations that mutate state. First argument is a mutable draft. |
| `invariants` | No | Self-contained business rules. Checked on `create()` and after every action. |
| `computed` | No | Derived getters. Each key becomes a property on the entity instance. |

> *One of `schema`, `type`, or `validate` is required. Pick the one that matches your validator.

## Instance API

| Member | Type | Description |
|---|---|---|
| `Entity.create(data)` | Entity instance | Static factory. Validates input through schema/type/validate, returns entity. |
| `entity.id` | `string` | The entity's identity. |
| `entity.props` | `Readonly<TProps>` | Frozen snapshot of current state. |
| `entity.actions.*` | `(...args) => void` | Action methods. Each mutates state through a shallow copy. |
| `entity.equals(other)` | `boolean` | Identity comparison by `id`. |

## How state mutation works

When you call an action, the library:

1. Reads current state from an internal `WeakMap`
2. Creates a **shallow copy** (`{ ...current }`)
3. Passes the copy to your action function — you mutate it freely
4. Checks invariants (if defined)
5. Writes the copy back to the `WeakMap`

```ts
// Your action
deactivate(state) {
  state.status = 'inactive';
  state.deactivatedAt = new Date();
}
// The library swaps the draft in — atomic from your perspective
```

> State never lives on `this`. It's stored in a `WeakMap` keyed by the entity instance.
> No public fields, no accidental exposure through iteration or logging.
> [Source: entity.ts:152-217](https://github.com/kaizin-lab/sotajs-ddd/blob/main/src/entity.ts)

## Why can't I mutate `props` directly?

```ts
user.props.bio = 'x'; // TypeError: Cannot assign to read only property
```

> `props` returns `deepFreeze({ ...current })` — a recursively frozen copy.
> Every read creates a new snapshot. The original state is untouched.
> This is not a convention — it's a physical constraint of the runtime.
> [Source: entity.ts:112-115](https://github.com/kaizin-lab/sotajs-ddd/blob/main/src/entity.ts)

## Invariants

Entity invariants are **self-contained** business rules — they only validate the entity's
own state, never nested entities (there aren't any). Use an Aggregate for cross-entity consistency.

```ts
const User = createEntity({
  schema: z.object({
    id: z.string(),
    name: z.string(),
    status: z.enum(['active', 'inactive']),
    deactivatedAt: z.date().optional(),
  }),
  actions: {
    deactivate(state) {
      state.status = 'inactive';
      state.deactivatedAt = new Date();
    },
  },
  invariants: [
    (props) => {
      if (props.status === 'inactive' && !props.deactivatedAt)
        throw new Error('Inactive user must have deactivatedAt');
    },
  ],
});

const user = User.create({ id: '1', name: 'Alice', status: 'active' });
user.actions.setName('Alicia');         // auto-setter — invariant holds
user.actions.deactivate();              // action — sets both fields, invariant holds
user.actions.setName('');               // auto-setter — passes here, but add your own rule
```

> Invariants are checked on `create()` and after **every** action — auto-setters included.
> If an invariant throws, the state is **not modified** — the draft copy is discarded.
> Keep invariants pure and synchronous: no I/O, no async, no network calls.
> [Source: entity.ts:184-188, 226-230](https://github.com/kaizin-lab/sotajs-ddd/blob/main/src/entity.ts)

## Auto-setters

For every non-`id` field in your schema, the library generates a `setField(value)` method:

```ts
// Schema: { id, username, bio }
user.actions.setUsername('new-name'); // generated automatically
user.actions.setBio('new-bio');       // generated automatically
```

> How? The library introspects `schema.shape` at definition time.
> Every key except `id` gets a setter. For ArkType or `validate` callbacks,
> auto-setters are discovered by calling `validate({ id: "any" })` and
> inspecting the returned keys. [Source: entity.ts:83-124](https://github.com/kaizin-lab/sotajs-ddd/blob/main/src/entity.ts)

## Validator-agnostic

Three equivalent ways to define the same entity:

```ts
// Zod / Valibot
createEntity({ schema: z.object({ id: z.string(), name: z.string() }), actions: {...} })

// ArkType
createEntity({ type: type({ id: 'string', name: 'string' }), actions: {...} })

// Plain function
createEntity({ validate: (data: unknown) => data as { id: string; name: string }, actions: {...} })
```

> The library only cares that your validator has a `.parse()` method (schema),
> is callable (type), or is a function (validate). Everything else is your choice.

---

## Pitfalls

- **Computed values are not cached.** They recalculate on every access. Keep them fast.
- **State is shallow-copied during actions.** If your state has nested objects, the draft shares references with the original for unchanged properties. `deepFreeze` on `.props` prevents external mutation.
- **Auto-setters skip when `validate({ id: "any" })` throws.** Define explicit setters in `actions` for strict validators.
