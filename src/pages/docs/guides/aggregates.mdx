An Aggregate is a cluster of entities treated as a unit for data changes.
Its job: ensure invariants hold after every mutation, no matter what.

## 1. Start with the entities inside

```ts
const OrderItemSchema = z.object({
  productId: z.string().uuid(),
  quantity: z.number().positive(),
  price: z.number().positive(),
});
const OrderItem = createEntity({ schema: OrderItemSchema, actions: {} });
type OrderItem = ReturnType<typeof OrderItem.create>;

const CustomerInfoSchema = z.object({
  name: z.string(),
  email: z.string().email(),
});
const CustomerInfo = createEntity({ schema: CustomerInfoSchema, actions: {} });
```

❌ **Skipping entity creation for nested objects:**

```ts
const OrderSchema = z.object({
  id: z.string().uuid(),
  items: z.array(z.object({ productId: z.string(), quantity: z.number(), price: z.number() })),
  // Plain object — no identity, no behavior, no auto-setters
});
```

> Plain objects inside an aggregate are just data. Entities give you `.equals()`,
> auto-setters, typed actions — even for nested items. Use `createEntity` for
> any object with its own identity.

## 2. Wire into createAggregate

```ts
import { createAggregate } from '@sotajs/ddd';

const OrderSchema = z.object({
  id: z.string().uuid(),
  status: z.enum(['pending', 'confirmed', 'shipped']),
  amountPaid: z.number(),
  items: z.array(OrderItemSchema),
  customerInfo: CustomerInfoSchema,
});

const Order = createAggregate({
  name: 'Order',
  schema: OrderSchema,
  entities: {
    items: OrderItem,
    customerInfo: CustomerInfo,
  },
  invariants: [
    (props) => {
      if (props.items.length === 0)
        throw new Error('Order must have at least one item');
    },
    (props) => {
      if (props.status === 'confirmed' && props.amountPaid <= 0)
        throw new Error('Cannot confirm unpaid order');
    },
  ],
  actions: {
    confirm(state) {
      if (state.status !== 'pending') throw new Error('Already processed');
      state.status = 'confirmed';
    },
    ship(state) {
      if (state.status !== 'confirmed') throw new Error('Not confirmed');
      state.status = 'shipped';
      return { event: { aggregateId: state.id, timestamp: new Date() } };
    },
  },
});
type Order = ReturnType<typeof Order.create>;
```

## 3. Invariants — required, run on every mutation

`invariants` is required. TypeScript won't compile without it.
All invariants run after every action — they don't guess which ones are "relevant":

```ts
order.actions.confirm(); // runs all invariants, including "must have at least one item"
```

❌ **Empty invariants array:**

```ts
invariants: [],
```

> This compiles but gives zero protection. If you don't have cross-field consistency
> rules, use `createEntity` instead of `createAggregate`. An aggregate without
> invariants is an entity with extra ceremony.

## 4. Nested entities — auto-hydrated

Entities declared in `entities` are auto-hydrated at creation:

```ts
const order = Order.create({
  id: '...', status: 'pending', amountPaid: 0,
  items: [
    { productId: '...', quantity: 2, price: 50 },
    { productId: '...', quantity: 1, price: 100 },
  ],
  customerInfo: { name: 'Alice', email: 'a@test.com' },
});

// items are real entity instances:
order.state.items[0].props.quantity; // 2
order.state.items[0].equals(order.state.items[1]); // false — different entities

// customerInfo is a real entity:
order.state.customerInfo.actions.setName('Alicia');
```

> On `.props` read, entities are dehydrated back to plain objects — ready for `JSON.stringify()`.
> Use `.state` when you need hydrated entity instances.

❌ **Mutating nested entities directly without actions:**

```ts
order.state.items[0].props.quantity = 5; // TypeError — props is frozen
```

> Use the entity's own actions or auto-setters: `order.state.items[0].actions.setQuantity(5)`.

## 5. Domain events — pull, don't push

Actions return `{ event }`. Events accumulate in a queue. You decide when to dispatch:

```ts
order.actions.ship();
const events = order.getPendingEvents();
// [{ aggregateId: '...', timestamp: ... }]

// Now dispatch — your event bus, your rules
await eventBus.publish(events);
```

❌ **Ignoring events:**

```ts
actions: {
  ship(state) {
    state.status = 'shipped';
    // No event returned — consumers don't know this happened
  },
}
```

> If downstream systems need to know about the mutation, return an event.
> Pull-based means no hidden side effects — if you don't pull, nothing fires.

❌ **Auto-dispatching inside the action:**

```ts
actions: {
  ship(state) {
    state.status = 'shipped';
    eventBus.publish({ ... }); // ❌ side effect inside the domain layer
  },
}
```

> The domain layer produces events. The application layer dispatches them.
> Keeping them separate makes the aggregate testable without mocking an event bus.

## 6. Aggregate vs Entity — the real difference

| | Entity | Aggregate |
|---|---|---|
| `invariants` | ❌ | ✅ Required |
| Cross-field rules on every mutation | ❌ | ✅ |
| Nested entities | Plain objects | Auto-hydrated instances |
| Domain events | ❌ | ✅ Pull-based queue |
| Auto-setters | ✅ | ❌ |
| Use when | Identity + behavior, no cross-field rules | Invariants must hold after every mutation |

❌ **Using Aggregate when Entity is enough:**

```ts
// User has no cross-field invariants — Entity is the right tool
const User = createAggregate({
  name: 'User',
  schema: UserSchema,
  invariants: [], // empty — no cross-field rules
  actions: { verifyEmail(state) { ... } },
});
```

> `createAggregate` exists for one reason: the `invariants` guarantee. Without
> invariants, `createEntity` is simpler and gives you auto-setters.

❌ **Using Entity when you need invariants:**

```ts
// Order has cross-field rules — Aggregate is the right tool
const Order = createEntity({
  schema: OrderSchema,
  actions: {
    confirm(state) {
      // Forgot to check amountPaid? Bug.
      state.status = 'confirmed';
    },
  },
});
```

> Entity can't enforce "order total must match item sum" after every action.
> You'd need to duplicate the check in every action. Aggregate runs invariants
> automatically — one definition, enforced everywhere.
