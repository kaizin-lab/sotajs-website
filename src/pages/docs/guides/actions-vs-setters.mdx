Every field in your Zod schema gets a free auto-setter: `setField(value)`.
You don't write them — the library generates them. The question is: when do you
write a custom action instead?

## The rule

> **If your action body is a single field assignment, delete it. An auto-setter already exists.**

❌ **Bad:**

```ts
actions: {
  updateBio(state, bio: string) {
    state.bio = bio; // ← auto-setter does this
  },
  setName(state, name: string) {
    state.name = name; // ← auto-setter does this
  },
  setEmail(state, email: string) {
    state.email = email; // ← auto-setter does this
  },
}
```

All three are redundant. The user can already call `user.actions.setBio(...)`,
`user.actions.setName(...)`, `user.actions.setEmail(...)`.

✅ **Good — action with a reason to exist:**

```ts
actions: {
  verifyEmail(state) {
    if (state.emailVerified) throw new Error('Already verified');
    state.emailVerified = true;
  },
  addRole(state, role: string) {
    if (!VALID_ROLES.includes(role)) throw new Error(`Unknown role: ${role}`);
    if (state.roles.includes(role)) throw new Error(`Already has role: ${role}`);
    state.roles.push(role);
  },
}
```

---

## When to write an action

| Situation | Action or auto-setter? |
|---|---|
| Set a single field to a given value | Auto-setter (`setField`) |
| Validate before setting | Action |
| Set multiple fields together | Action |
| Append to an array with validation | Action |
| State machine transition (`pending` → `confirmed`) | Action |
| Guard clause or conditional logic | Action |
| Return a domain event | Action (Aggregate only) |

## Auto-setters are typed

They inherit types from your Zod schema:

```ts
const UserSchema = z.object({
  id: z.string().uuid(),
  username: z.string(),
  age: z.number(),
});

// These are auto-generated and fully typed:
user.actions.setUsername('alice'); // string → ✓
user.actions.setAge(25);           // number → ✓
user.actions.setAge('25');         // TypeScript error
```

---

## Auto-setters accept the field's Zod type

If your schema narrows the type, the auto-setter reflects it:

```ts
const OrderSchema = z.object({
  id: z.string(),
  status: z.enum(['pending', 'confirmed', 'shipped']),
});

// setStatus only accepts valid status values:
order.actions.setStatus('confirmed'); // ✓
order.actions.setStatus('cancelled'); // TypeScript error
```

## Common anti-patterns

❌ **Wrapping an auto-setter to rename it:**

```ts
actions: {
  changeBio(state, bio: string) {
    state.bio = bio; // Just call setBio() instead
  },
}
```

If you want a different name — use the auto-setter. `user.actions.setBio(bio)` is the canonical name.

❌ **Action that does nothing but assign:**

```ts
actions: {
  setActive(state) {
    state.active = true; // user.actions.setActive(true) is auto-generated
  },
}
```

❌ **Validating in an action what Zod already validates:**

```ts
const schema = z.object({
  email: z.string().email(), // Zod validates this
});

actions: {
  setEmail(state, email: string) {
    if (!email.includes('@')) throw ...; // redundant — Zod already checked
    state.email = email;
  },
}
```

> Zod validates on `.create()`. Auto-setters don't re-validate (they bypass the schema).
> If you need re-validation on set, write an action that calls `schema.parse()`.

---

## When auto-setters are NOT generated

Auto-setters are discovered by calling `validate({ id: "any" })` and inspecting the returned keys.
If your validator rejects `{ id: "any" }` (strict validators that require all fields),
auto-setters are skipped. In that case, define setters explicitly in `actions`.

With Zod/Valibot schemas (recommended), auto-setters are generated from `schema.shape` —
no trial validation needed. They always work.
