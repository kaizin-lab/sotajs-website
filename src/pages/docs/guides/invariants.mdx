## Where invariants live

There are four places invariants can live. Two are design-time guarantees (automatic),
two are developer discipline:

| Location | Enforcement | Scope | Example |
|---|---|---|---|
| Zod schema | Automatic (parser) | Single field | `z.string().email()` |
| Entity `invariants` | Automatic (every action) | Self-contained cross-field | "Inactive user must have deactivatedAt" |
| Aggregate `invariants` | Automatic (every action) | Cross-entity cross-field | "Total must match sum of item prices" |
| Action guard clause | Manual (per-action) | Per-action precondition | "Can't ship unconfirmed order" |

Zod catches `"notanemail"`. Entity invariants catch `status=inactive` without `deactivatedAt`.
Aggregate invariants catch `Order.total ≠ sum(OrderItem.price × qty)`. Action guards catch
"this action shouldn't be called right now."

## Entity invariants

Entity now has an optional `invariants` field. These are **self-contained** — they only
validate the entity's own props, never nested entities (there aren't any).

```ts
const User = createEntity({
  schema: z.object({
    id: z.string(),
    name: z.string(),
    status: z.enum(['active', 'inactive']),
    deactivatedAt: z.date().optional(),
  }),
  actions: {
    deactivate(state) {
      state.status = 'inactive';
      state.deactivatedAt = new Date();
    },
  },
  invariants: [
    (props) => {
      if (props.status === 'inactive' && !props.deactivatedAt)
        throw new Error('Inactive user must have deactivatedAt');
    },
  ],
});
```

> Invariants run on `create()` and after **every** action — auto-setters included.
> If an invariant throws, the draft is discarded and state is unchanged.

❌ **Action guard for cross-field rules (the old way):**

```ts
actions: {
  deactivate(state) {
    state.status = 'inactive';
    state.deactivatedAt = new Date();
    // Each action must remember to set both fields — no design-time guarantee
  },
  reactivate(state) {
    state.status = 'active';
    state.deactivatedAt = undefined;
    // Did we remember? If we forget deactivatedAt, only hope catches it.
  },
}
```

> Without invariants, every action that touches `status` must remember `deactivatedAt`.
> With invariants, you write the rule once — the library enforces it on every mutation.

## Aggregate invariants

`invariants` on Aggregate is **required**. It runs at `.create()` and after
**every** action — all invariants, not just "relevant" ones.

The key difference from Entity: Aggregate invariants can span **nested entities**.
When `Order.items` hydrates into `OrderItem[]` entity instances, invariants can
validate relationships between the root and its children:

```ts
const Order = createAggregate({
  name: 'Order',
  schema: OrderSchema,
  entities: { items: OrderItem },
  invariants: [
    (props) => {
      const itemTotal = props.items.reduce((sum, i) => sum + i.price * i.quantity, 0);
      if (Math.abs(props.total - itemTotal) > 0.01)
        throw new Error('Total must match sum of item prices');
    },
  ],
  actions: {
    confirm(state) {
      if (state.status !== 'pending') throw new Error('Already processed');
      state.status = 'confirmed';
    },
  },
});
```

> This is the aggregate's unique guarantee: invariants that span **multiple entities**
> within a transactional boundary. Entity invariants can't do this — they only see
> their own props.

## Entity vs Aggregate for invariants

| | Entity | Aggregate |
|---|---|---|
| Single-field validation | Zod schema | Zod schema |
| Self-contained cross-field rule | `invariants` (optional) | `invariants` (required) |
| Cross-entity rule | ❌ Not possible | `invariants` (required) |
| Action precondition | Guard in action | Guard in action |

Use **Entity invariants** when the rule involves only the entity's own fields
(`status=inactive ⇒ deactivatedAt`). Use **Aggregate invariants** when the rule
spans the root and its children (`total = sum(items)`).

❌ **Aggregate with empty invariants:**

```ts
createAggregate({
  name: 'User',
  schema: UserSchema,
  invariants: [], // empty — use createEntity instead
  actions: { ... },
})
```

> If you have no cross-entity invariants and no nested entities — use `createEntity`.
> Aggregate exists for the cross-entity consistency guarantee.

## Invariants must be synchronous

❌ **Async invariant:**

```ts
invariants: [
  async (props) => {
    const exists = await db.users.count({ where: { email: props.email } });
    if (exists) throw new Error('Email taken');
  },
]
```

> Uniqueness checks belong in the use case before entity creation, not in invariants.
> Invariants validate the entity's own state — not the outside world.

## What you've learned

**Four locations for validation.** Zod for format, Entity invariants for self-contained
cross-field rules, Aggregate invariants for cross-entity rules, action guards for
per-action preconditions.

**Invariants run on every action.** All invariants, every time — not just the ones
that seem "relevant." Both Entity and Aggregate enforce this guarantee. The difference
is scope: Entity invariants see only the entity's own props; Aggregate invariants
span nested entities.

**Invariants are synchronous and pure.** No database calls, no HTTP. Validate the
entity's own state. External checks (uniqueness, existence) belong in the use case.
