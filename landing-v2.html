Tactical DDD for Node.js / TypeScript

Tactical DDD was designed
for a runtime Node doesn't provide

Nominal typing, real encapsulation, synchronous transactional consistency — DDD patterns assume a Java/C# runtime. In Node you pay for every invariant with manual infrastructure: brand types, hand-rolled repositories, optimistic locking. Teams cut corners — keeping the vocabulary, losing the guarantees.

The gap

Five reasons DDD is expensive in Node — and what we fix

01

Structural typing sabotages Value Objects

Two {value: string} are interchangeable even when one is Email and one is Username. Brand types are a manual hack every team writes differently — or skips.

02

No runtime encapsulation

JSON.stringify and ORM mapping bypass privacy. Private fields (#) exist but serialization flattens aggregates into plain objects. Invariants don't survive the persistence boundary.

03

async/await breaks transactional consistency

Domain events in Java are collected during one synchronous transaction. In Node, parallel requests to one aggregate race. Optimistic locking is rarely built — and rarely done.

04

No ORM supports rich domain models

Prisma generates anemic DTOs. Drizzle is a SQL builder. Every Repository pattern requires custom mapping. The cost of "clean" DDD is disproportionate for medium projects.

05

Culture: transformations, not invariants

Node grew from Express middleware and functional pipelines — thinking in data flows, not in consistency boundaries. DDD patterns feel like cargo-cult without culture fit.

→

We close the gap

@sotajs/ddd gives you compile-time branded IDs, runtime encapsulation (WeakMap + deepFreeze), invariant enforcement on every action, and validator-agnostic schemas — four functions, zero lock-in.

The cost of doing it right

Rich domain model. Two ways to write it.

Manual — 45 lines, never written 
class User {
  private bio: string;
  private username: string;
  
  constructor(private props: UserProps) {
    this.bio = props.bio;
    this.username = props.username;
  }

  updateBio(bio: string) {
    if (bio.length > 500)
      throw new Error("Too long");
    this.bio = bio;
  }
  updateUsername(name: string) {
    if (!name) throw new Error("Required");
    this.username = name;
  }

  get props(): Readonly<UserProps> {
    return Object.freeze({...this});
  }
  equals(other: User): boolean {
    return this.id === other.id;
  }
  // ×20 entities = 900 lines
  // Nobody writes this.
}
@sotajs/ddd — 3 config fields 
const User = createEntity({
  schema: UserSchema,
  actions: {
    updateBio(state, bio) { state.bio = bio; },
    updateUsername(state, n) { state.username = n; },
  },
  computed: {
    displayName(props) {
      return props.username;
    },
  },
});
// deepFreeze, auto-setters,
// equals, type inference —
// all automatic.

Full example

One declaration. What you get.

user-profile.entity.ts 
import { z } from 'zod';
import { createEntity } from '@sotajs/ddd';

const UserProfileSchema = z.object({
  id: z.string().uuid(),
  username: z.string(),
  bio: z.string().optional(),
});
type UserProfileProps = z.infer<typeof UserProfileSchema>;

export const UserProfile = createEntity({
  schema: UserProfileSchema,
  actions: {
    updateUsername(state, name: string) {
      state.username = name;
    },
    updateBio(state, bio: string) {
      state.bio = bio;
    },
  },
  computed: {
    displayName(props) { return props.username; },
  },
});
export type UserProfile =
  ReturnType<typeof UserProfile.create>;
01

Schema = types = validation

z.infer gives you props type. ReturnType gives you entity type. One source of truth, zero drift.

02

Actions — the only path to mutation

props is a frozen snapshot. entity.props.bio = "x" throws TypeError. Changes only through named, typed actions.

03

Auto-setters from schema

Every field except id gets a typed setField() method. Generated from schema.shape — no manual code.

04

Computed properties, not function calls

profile.displayName is a getter. Defined once in config, reads like native data.

05

Identity by id, not reference

a.equals(b) compares by id. Two instances from different queries — same entity.

06

Zero runtime dependencies

No Immer, no DI, no ORM. Works with Zod, ArkType, Valibot, or a plain function you write.

In use

From declaration to use case

The entity is a typed, safe object you use in commands — no framework, no ORM, no DI.

// Schema validates on create
const profile = UserProfile.create({

// Use in a command — actions are typed
export const updateProfileCommand = async (
  userId: string, bio: string,
) => {
  const profile = await profiles.findById(userId);
  
  profile.actions.updateBio(bio);
  await profiles.save(profile);
  
  return {
    id: profile.id,
    displayName: profile.displayName,
  };
};

Your stack

Four functions. Not a framework.

→

Works inside NestJS — or anything

Your controllers, DI, and modules stay as they are. The library lives in the domain layer. No decorators, no base classes.

→

No new vocabulary

You already write Zod schemas for DTOs. That's the schema. Actions are functions. Computed are getters. The library removes the boilerplate between them.

→

One module at a time

Replace interface Order with createAggregate in one service. The app doesn't notice. Roll back if it doesn't fit.

→

When you don't need it

CRUD with no business rules? z.object + interface is enough. The library earns its place when invariants change and business logic outgrows services.
