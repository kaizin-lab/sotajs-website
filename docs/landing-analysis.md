# sotajs/ddd — аналитический фундамент для лендинга

## Часть 1: Тактический DDD как база

### Тактический DDD: что он предписывает

Четыре паттерна, которые не про «философию», а про конкретные гарантии в коде:

| Паттерн | Что даёт | Без него |
|---|---|---|
| **Entity** | Поведение живёт с данными. Мутация — только через именованные методы. | `user.bio = x` где угодно. Ищи кто поменял. |
| **Aggregate** | Транзакционная граница. Инварианты проверяются на каждом изменении. | `order.total` расходится с `order.items`. Инвариант был в одном сервисе, забыли в другом. |
| **Value Object** | Неизменяемость. Равенство по значению. | `money1 === money2` — false, хотя сумма и валюта одинаковые. |
| **Domain Events** | Явный факт «что случилось». Отделяет сайд-эффекты от мутации. | `sendEmail()` внутри `addItem()`. Нельзя протестировать одно без другого. |

Это не «методология для энтерпрайза». Это инженерные гарантии: ты не можешь сломать инвариант, потому что код не даст. Ты не можешь забыть валидацию, потому что она встроена в `create()`. Ты не можешь пропустить событие, потому что оно возвращается из действия.

### Богатые модели: почему это база

**Проблема не в том что разработчики не знают DDD. Проблема в том что «правильный» код дороже «быстрого».**

```typescript
// БЫСТРО (anemic):
user.bio = newBio;                    // 1 строка
await db.users.update(user.id, user); // 1 строка

// ПРАВИЛЬНО (rich, без библиотеки):
class User {
  private bio: string;
  updateBio(newBio: string) {          // метод вместо присваивания
    if (newBio.length > 500) throw ... // инвариант
    this.bio = newBio;
  }
  get props(): Readonly<UserProps> {   // защитная копия
    return Object.freeze({...this});
  }
  equals(other: User) {                // identity
    return this.id === other.id;
  }
}
```

Разработчик ЗНАЕТ что богатая модель правильнее. Но плата: +15 строк на каждое поле, на каждый инвариант. И это только Entity — без агрегатов, событий, репозиториев.

Результат: команда выбирает anemic. Не потому что не понимает ценность. Потому что **цена правильности выше цены быстрого кода**.

### createEntity: что он делает с этой проблемой

`createEntity` не учит DDD. Он убирает трение:

```typescript
// Та же богатая модель. 3 поля конфига вместо 40 строк бойлерплейта.
const User = createEntity({
  schema: UserSchema,        // ← уже написал для DTO
  actions: { updateBio },    // ← только значимое поведение
  computed: { displayName }, // ← derived, без get-болванки
});
```

**Что исчезло:**

| Бойлерплейт | Кто убрал |
|---|---|
| Приватное поле + геттер | `WeakMap`-стейт внутри фабрики |
| `Object.freeze({...this})` | `deepFreeze` при каждом `.props` |
| `equals()` по id | Автоматически из `id` поля |
| Сеттеры для каждого поля | Авто-генерация из `schema.shape` |
| Дублирование типа и валидации | `z.infer<typeof schema>` = тип = валидация |
| `class User { ... }` + `new User()` | Фабрика возвращает класс |

**Что осталось разработчику:**
- Описать данные — Zod-схема (он и так это делает)
- Назвать действия — `actions: { updateBio }`
- Назвать computed — `computed: { displayName }`

Три вещи, которые он В ЛЮБОМ СЛУЧАЕ должен был бы определить: форму данных, операции над ними, производные свойства. Библиотека не добавляет концептов — только убирает механику.

### DX без когнитивной сложности

Ключевое утверждение: **библиотека не требует понимать DDD чтобы получить выгоду от DDD.**

Разработчик думает: «Я описал схему. Я описал что можно менять. Я описал что computed. Библиотека дала мне typed entity с защищённым стейтом. Мне не нужно знать что это "rich domain model" — я просто получил безопасный код быстрее чем написал бы unsafe.»

Это и есть определение хорошей абстракции: **ты получаешь правильную архитектуру как побочный эффект использования инструмента, а не как результат следования методологии.**

## Часть 2: Тактический DDD за пределами Entity

### Что ещё предлагает тактический DDD

Классический тактический DDD (Evans) — это система гарантий, а не один паттерн. Каждый закрывает конкретный класс ошибок:

| Паттерн | Гарантия | Класс ошибок, который предотвращает |
|---|---|---|
| **Entity** | Мутация только через именованные методы | Неизвестно кто и где поменял поле |
| **Aggregate** | Инварианты после каждого изменения | Рассогласованность данных внутри границы |
| **Value Object** | Равенство по значению, неизменяемость | `{amount:5} !== {amount:5}`, случайная мутация |
| **Domain Events** | Явный факт «что произошло» | Сайд-эффекты приклеены к мутации |
| **Repository** | Абстракция персистенса для агрегата | Доменный код знает про SQL/ORM |
| **Factory** | Инкапсуляция сложного создания | Конструктор с 10 параметрами |
| **Domain Service** | Безстатусная операция над несколькими агрегатами | Логика не принадлежит ни одному Entity |
| **BrandedId** (TS-специфично) | `UserId ≠ ProductId` на уровне компилятора | Перепутал ID местами — компилятор молчит |

### Что из этого реализует @sotajs/ddd

| Паттерн | Реализация | Что даёт сверх «написать руками» |
|---|---|---|
| Entity | `createEntity` | schema-agnostic валидация, автосеттеры, deepFreeze, computed |
| Aggregate | `createAggregate` | invariants НЕ опциональны, nested entity hydration, очередь доменных событий |
| Value Object | `createValueObject` | structural equality (JSON compare), deepFreeze, optional actions |
| Domain Events | внутри `createAggregate` | pull-based: событие возвращается из action, диспатчит use case |
| BrandedId | `createBrandedId` | compile-time брендирование, валидация, JSON-сериализация |
| **Repository / Factory / Domain Service** | НЕ реализованы | Намеренно — это слой приложения/инфраструктуры, а не домена |

### Ключевые дизайн-решения библиотеки

#### 1. Aggregate: invariants обязательны

```typescript
createAggregate({
  invariants: [...], // ← required field, не optional
})
```

Если у тебя нет инвариантов — тебе не нужен Aggregate, используй Entity. Библиотека не даёт создать «пустой» агрегат. Это enforces ключевой принцип DDD: **агрегат существует чтобы защищать бизнес-правила в рамках транзакционной границы.**

#### 2. Nested entity hydration

```typescript
createAggregate({
  entities: {
    customerInfo: CustomerInfo, // ← Entity-класс
  },
})

// При создании сырые данные авто-оборачиваются:
order.state.customerInfo // ← экземпляр CustomerInfo, не plain object
order.state.customerInfo.actions.updateName(...) // ← rich model в действии
```

Это «богатая модель» на практике: дочерние сущности внутри агрегата — не просто данные, а полноценные объекты с поведением. Библиотека делает hydration автоматически.

#### 3. Domain events: pull, not push

```typescript
actions: {
  ship: (state) => {
    state.status = 'shipped';
    return { event: new OrderShipped(state.id) };
  },
}

// События накапливаются в очереди:
order.actions.ship();
const events = order.getPendingEvents(); // ← вызывающий код решает что делать
```

В отличие от типичного `this.eventBus.emit(...)` внутри метода, здесь:
- Действие чистое — только мутирует стейт
- Событие — возвращаемое значение, не сайд-эффект
- Use case решает когда и как диспатчить события
- Тестирование: проверил что событие вернулось — не нужен mock event bus

#### 4. Value Object: мутабельный внутри, иммутабельный снаружи

```typescript
const Money = createValueObject({
  schema: MoneySchema,
  actions: {
    add: (state, other) => { state.amount += other.amount; },
  },
});

money.actions.add(anotherMoney); // ← внутренняя мутация через action
money.props // ← замороженная копия
```

Чистый DDD сказал бы «VO неизменяем, `add` возвращает новый экземпляр». Но `money.actions.add(...)` читается естественнее чем `money = money.add(...)` с переприсваиванием. `deepFreeze` на `.props` гарантирует что снаружи никто не мутирует.

#### 5. Что НЕ реализовано — и почему

| Не реализовано | Причина |
|---|---|
| Repository | Принадлежит слою инфраструктуры. Библиотека не диктует как хранить данные. |
| Domain Service | Это функция в application-слое. Библиотека даёт доменные объекты, оркестрация — на разработчике. |
| Factory | `.create()` покрывает 90% случаев. Сложные фабрики — доменная логика, пиши сам. |
| Specification | Паттерн хороший, но implementation зависит от контекста. Легко добавить поверх. |

### Главный вывод для позиционирования

**Библиотека даёт не «набор DDD-паттернов», а гарантии, которые без неё держатся только на дисциплине.**

| Гарантия | Без библиотеки | С библиотекой |
|---|---|---|
| Инварианты после мутации | Разработчик помнит → проверил | `invariants: [...]` → проверяется всегда |
| Неизменяемость props | `Object.freeze()` в каждом геттере | `deepFreeze` автоматически |
| Структурное равенство | `JSON.stringify(a) === JSON.stringify(b)` | `a.equals(b)` |
| Типизированные ID | `type UserId = string` (стирается) | `createBrandedId({ brand })` (compile-time) |
| Домен не знает про инфраструктуру | Архитектурная дисциплина | События возвращаются, не отправляются |
